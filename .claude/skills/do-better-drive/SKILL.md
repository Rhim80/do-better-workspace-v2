# Do Better Drive — 7단계 프로젝트 오케스트레이터

프로젝트 하나를 **목적 → 완료**까지 7단계 게이트로 미는 비개발자용 방법론. 개발자용 gstack(Garry Tan, YC)을 이림 규칙으로 이식한 것. 이 스킬은 **지휘자** — 어느 단계를 할지 판단해 그 단계의 상세 지침 파일을 열어 수행한다. 상세 rigor는 전부 `references/`에 있다.

## 세 축 (방법론의 뼈대 — 모든 단계에 적용)

1. **앞 3단계는 만들기 금지** — 프레임·범위·계획까지 아무것도 안 만든다. 방향이 서기 전엔 손을 못 대게 막는다.
2. **일을 3층으로 갈라 완성도를 다르게** — AI가 하는 일(초안·생산·정리)은 풀버전으로, 사람이 하는 일(취향·핵심 판단·비가역 결정)은 초점을 주고, 몸으로 하는 일(요리·현장)은 병목으로 인정.
3. **되돌림이 정상** — 검토(⑤)에서 큰 문제가 나오면 앞 단계로 되돌린다. 실패가 아니라 이 방법이 원래 하는 일.

**재구현 금지** — 생산·발행·축적은 기존 스킬(콘텐츠 스킬·ghost-publish·wiki-ingest 등)에 위임한다. 각 단계는 지휘·검사·게이트 역할이지 도구를 새로 짜지 않는다.

## Step 0 — 어느 단계인지 판정 (가장 중요)

아래 순서로 이번에 수행할 단계를 정한다.

1. **참가자 표현이 특정 단계를 지목**하면 그 단계로 (아래 매핑 표).
2. 지목이 없으면 **활성 프로젝트의 `10-projects/<프로젝트>/progress.md`를 열어 `status:`로 이어갈 단계를 정한다** (status → 다음 단계 표). 어느 프로젝트인지 불분명하면 `AskUserQuestion`으로 고르게 한다.
3. progress.md가 없고 **새 프로젝트면 ① 프레임**부터.
4. **애매하면 추측하지 말고 묻는다.**

### 트리거 → 단계 → 읽을 파일

| 참가자 표현 | 단계 | 읽을 파일 |
|---|---|---|
| 새 프로젝트 시작 · 이거 할까 말까 · 프로젝트 프레임 · 목적부터 잡자 | ① 프레임 | `references/1-frame.md` |
| 범위 도전 · 이거 될 일인가 · 실현성 점검 · 시장성 점검 | ② 될 일인가 | `references/2-scope.md` |
| 계획 짜자 · 구조 잡자 · 실행계획 | ③ 계획 | `references/3-plan.md` |
| 생산 시작 · 만들기 시작 · 변형 뽑아 비교 · 초안 여러 개 | ④ 생산 | `references/4-make.md` |
| 검토 · 리뷰 · 이거 됐나 봐줘 · 합격인가 · 냉정하게 봐줘 | ⑤ 냉정한 검토 | `references/5-review.md` |
| 내보내기 · 발행하자 · 이제 내보내 · 출간 · 모객 시작 · 전달하자 | ⑥ 내보내기 | `references/6-ship.md` |
| 돌아보기 · 회고 · 이번 프로젝트 정리 · 배운 것 뽑아줘 · retro | ⑦ 돌아보기 | `references/7-retro.md` |

### progress.md `status:` → 이어갈 단계

`status` 값은 **마지막으로 통과한 단계**다. 이어가기는 그 다음 단계다.

| status | 다음에 할 단계 |
|---|---|
| (없음 / 새 프로젝트) | ① 프레임 |
| 프레임 | ② 될 일인가 |
| 범위 | ③ 계획 |
| 계획 | ④ 생산 |
| 생산 | ⑤ 냉정한 검토 |
| 리뷰 | ⑥ 내보내기 |
| 발행 | ⑦ 돌아보기 |
| 완료 | 닫힘 (이어질 새 판이면 ① 프레임으로 새 progress.md 시작) |
| 보류 | 막힌 단계 — 뭐가 안 섰는지 progress.md "실현성 점검"/"검토"에서 확인하고 참가자와 정한다 |

## 실행

정한 단계의 `references/N-*.md`를 **Read 해서 그 단계를 그 파일대로 온전히 수행한다.** 각 파일은 그 단계의 전체 지침(자세·질문·위험신호·판정 어휘·게이트·예시)을 담고 있다 — 요약하지 말고 그대로 따른다. 필요할 때만 읽고 인라인하지 않는다.

**되돌림**: 참조 파일이 "N단계로 되돌린다"고 하면, 그 단계의 `references/` 파일로 라우팅해 이어간다. 되돌림은 정상이다.

## 게이트 규율 (방법론 본질 — 지킬 것)

한 단계를 끝내면:
1. **progress.md를 갱신**한다 (그 단계가 채우는 칸 + "다음 한 걸음" + `status:` + 로그 한 줄).
2. **다음 단계를 안내**한다 (예: "프레임 섰어요. 다음은 ② 될 일인가.").
3. **멈춘다.** 참가자가 "다음" 할 때 이어간다 — **다음 단계를 자동으로 실행하지 않는다.** (게이트 사이의 멈춤이 이 방법의 핵심이다.)

## 공유 상태 — 표준 progress.md 형식 (모든 단계의 단일 출처)

프로젝트는 `10-projects/<프로젝트>/progress.md` 하나로 진행을 기록한다 (폴더 번호는 워크스페이스 Johnny Decimal 규칙). 표준 형식:

```markdown
---
type: progress
initiative: <프로젝트 이름>
mode: 수요형 | 창작형
status: 프레임    # 프레임 → 범위 → 계획 → 생산 → 리뷰 → 발행 → 완료 (막히면 보류)
updated: <YYYY-MM-DD>
---

# <프로젝트 이름> — 진행

## 목표          ← ①프레임: 목적 한 문장 + 대상 한 줄
## 실현성 점검    ← ②될 일인가
## 계획          ← ③계획
### 합격 기준     ← ③이 정함. ⑤검토가 이 기준으로 검사.
## 생산          ← ④생산: 뽑은 컨셉·확정한 것과 이유·생산물 위치
## 검토          ← ⑤냉정한 검토: 판정(합격/조건부/불합격)·합격 기준 대조·계획 이행 감사
## 지금 상태
## 다음 한 걸음   ← 항상 채워둔다. 이게 프로젝트를 굴리는 힘.
## 로그
- <YYYY-MM-DD> 프레임 확정 (drive-frame)
```

이미 있는 progress.md면 해당 칸과 "다음 한 걸음"만 갱신하고 로그에 한 줄 append.

## Reference 파일 (필요한 단계만 Read)

- `references/1-frame.md` — ① 프레임: 목적·대상을 캐물어 확정 (만들기 금지). 새 프로젝트 진입.
- `references/2-scope.md` — ② 될 일인가: 효과 실현성(닿는 규모 × 전환)을 근거로 검증.
- `references/3-plan.md` — ③ 계획: 접근 택1·구조·합격 기준·유통·3층 구분.
- `references/4-make.md` — ④ 생산: 변형 여러 개 → 비교 → 취향 택1 → 풀생산(기존 스킬 위임).
- `references/5-review.md` — ⑤ 냉정한 검토: 합격 기준 대조 + 계획 이행 감사 + 다렌즈 병렬 검토.
- `references/6-ship.md` — ⑥ 내보내기: 정한 경로로 발행(기존 스킬 위임) + 민감정보 점검 + 사람 확정.
- `references/7-retro.md` — ⑦ 돌아보기: 교훈 포착 → 회고·프로필·위키(wiki-ingest)로 복리 축적.
