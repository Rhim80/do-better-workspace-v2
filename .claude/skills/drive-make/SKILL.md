# Do Better Drive · 4단계 — 생산 게이트

> Do Better Drive 7단계 중 4단계. 3단계(drive-plan)가 "어떻게·무엇이 되면 합격인가"를 잠갔다면,
> 이 단계는 **처음으로 실제로 만든다.** 단 통짜 하나로 직진하지 않는다.
> 개발자용 gstack의 design-shotgun(변형 여러 개 비교)+design-html(확정본 풀생산)을 비개발 프로젝트에 맞춘 것.

## 이 스킬의 역할

Do Better Drive에서 **처음으로 만들기가 허용되는 단계.** (앞 3개는 만들기 금지였다.) 하지만 완성본 하나로 곧장 가지 않는다 — 서로 다른 변형 여러 개를 **가볍게 먼저** 뽑아 나란히 비교하고, 취향으로 하나를 확정한 뒤, **그 방향만** 풀버전으로 생산한다. 첫 삽부터 한 방향에 갇혀 엉뚱한 걸 열심히 완성하는 걸 막는다.

**핵심 순서: 변형 → 비교 → 확정 → 풀생산.** 건너뛰지 않는다.
**재구현 금지: 풀생산은 이미 있는 콘텐츠·산출 스킬을 부른다.** 여기서 생산 도구를 새로 짜지 않는다.

## 작동

### 0. 입력 읽기 + 준비 확인
- 3단계가 갱신한 `progress.md`의 **"계획"**(구조·순서·층 구분)과 **"합격 기준"**을 읽는다.
- **둘 중 하나라도 없으면** 아직 계획이 안 잠긴 것 → drive-plan(3단계)로 돌려보낸다. 여기서 즉석으로 계획 잡지 않는다.
- `CLAUDE.md` 프로필(특히 "판단 성향") · `08-registry`(있으면 참고).

### 1. 무엇을 만드나 + 어떤 스킬로 (판정)
- 이번에 만들 산출물이 무엇인지 계획에서 집는다 (예: 책 챕터 / 콘텐츠 한 편 / 과정 모듈 / 세션 흐름 / 제품·상세 / 랜딩).
- 그걸 **풀버전으로 만들 기존 스킬 후보**를 찾는다 — 이 워크스페이스·프로필에 있는 콘텐츠·산출 스킬을 쓴다. 맞는 스킬이 없으면 직접 만들되(뒤 5단계), 새 생산 스킬을 즉석에서 짜지 않는다.

### 2. 변형 여러 개 (가볍게)
- 완성본 하나 말고, **서로 다른 변형 2~3개**를 뽑는다. 단 지금은 **가볍게** — 컨셉·앞부분 샘플·스케치 수준이면 된다. (풀생산은 확정 뒤에 한 번만.)
- **진짜로 달라야 한다** — 각도·톤·구조 중 최소 하나가 실제로 갈려야 변형이다. 미묘하게 다른 셋은 변형이 아니라 같은 것 셋이다.
- 이건 주로 **AI가 하는 일** 층이라 싸다 → 부담 없이 여러 개 뽑는다. 단 **몸으로 만들어야 하는 산출물**(요리·실물 제작·현장 진행)은 변형을 싸게 못 찍는다 → 변형 수를 줄이고 그 사실을 그대로 말한다.

### 3. 나란히 비교
- 변형들을 **한자리에서 비교**한다 (표 또는 나란히 배치). 각 변형의 강점·약점, 그리고 **합격 기준에 비춰** 어디가 세고 어디가 약한지 짚어준다.
- Claude가 도장 찍어 하나로 미는 게 아니라, 고를 재료를 차려준다.

### 4. 취향으로 확정 (사람 몫)
- **`AskUserQuestion`으로 참가자가 하나를 고른다** (또는 "A의 이 부분 + B의 이 부분"으로 섞기).
- 이 선택이 **사람층의 핵심 판단** — 취향·중심 줄기라 AI가 대신 정하지 않는다. 참가자 프로필의 "판단 성향"이 있으면 참고하되, 최종 선택은 참가자가 한다.
- 고른 이유(취향)를 로그에 한 줄 남긴다 → 다음 생산 때 참고.

### 5. 풀생산 (기존 스킬 호출)
- 확정된 방향 하나를 **풀버전으로 생산한다.** 1단계에서 잡은 기존 콘텐츠·산출 스킬을 부른다 — 여기서 재구현하지 않는다.
- 계획의 층 구분대로 완성도를 건다:
  - **AI가 하는 일** — 풀버전으로. 커버리지·채널·마감 다 챙긴다. 여기선 덜 하지 않는다.
  - **사람이 하는 일** — 확정된 취향·핵심 판단을 반영한다.
  - **몸으로 하는 일** — 병목이면 병목대로 잡는다 (AI 일처럼 싸다고 착각하지 않는다).
- **합격 기준을 보며** 만들되, **자기 합격 도장은 찍지 않는다** — 검사는 5단계(냉정한 검토) 몫이다.

### 6. 핸드오프 (산출)
`progress.md`를 갱신한다:
- **"생산"** 칸에 기록 — 뽑은 변형(무엇이 갈렸나) / 확정한 것과 이유 / 부른 스킬 / **생산물이 어디 있는지(경로·링크)**
- **"다음 한 걸음"** 갱신
- frontmatter `status: 생산`, `updated` 갱신
- 로그에 `- <YYYY-MM-DD> 생산 (drive-make)` append
- 안내: **"생산했어요. 다음은 5단계 — 냉정한 검토."**

여러 산출물을 만드는 프로젝트면(예: 콘텐츠 여러 편) 이 스킬을 산출물마다 다시 돈다. 매번 변형 → 비교 → 확정 → 풀생산.

## 규칙 요약
- **통짜 직진 금지** — 변형 여러 개 → 비교 → 확정 → 풀생산. 순서를 건너뛰지 않는다.
- **변형은 가볍게, 진짜로 다르게** — 확정 전엔 컨셉·샘플 수준. 미묘한 차이 셋은 변형이 아니다.
- **확정은 사람 몫** — 취향 선택은 `AskUserQuestion`으로 참가자가.
- **재구현 금지** — 풀생산은 기존 콘텐츠·산출 스킬을 부른다.
- **층대로 완성도** — AI층 풀버전, 사람층 취향 초점, 몸노동 병목 인정.
- **합격 도장은 안 찍는다** — 합격 기준을 보며 만들되, 검사는 5단계 몫.
